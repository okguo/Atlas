# 一、文档目标

本文档用于描述 Atlas 的整体系统架构。

重点说明：

- Atlas 如何运行
- 系统由哪些核心模块组成
- 各模块之间如何协作
- 为什么采用当前架构
- 系统未来如何扩展

本文档不涉及：

- 具体代码实现

这些内容将在后续文档中分别说明。

# 二、架构目标

Atlas 是一个 AI 内容生产操作系统（Content Production Operating System）。

因此，系统架构必须满足以下目标：

## 2.1 可扩展

任何能力都应该可以新增。

例如：

- 新增 Agent
- 新增 Workflow
- 新增插件
- 新增发布平台

无需修改 Atlas Core。

---

## 2.2 可组合

任何 Agent 都应该能够自由组合。

例如：

Research Agent

↓

Script Agent

↓

Narration Agent

↓

Publish Agent

可以组成：

YouTube Workflow。

也可以组成：

公众号 Workflow。

---

## 2.3 可替换

Atlas 不绑定任何 AI 厂商。

任何能力都可以替换。

例如：

LLM

TTS

图片生成

视频生成

全部支持多个 Provider。

---

## 2.4 可恢复

任何任务都应该：

暂停

恢复

重试

重新执行

支持人工介入。

---

## 2.5 可观测

整个 Workflow 的每一步：

输入

输出

耗时

Token

日志

错误

全部可追踪。

---

# 三、整体架构

Atlas 采用：

> **Workflow + Agent + Plugin**

三层核心架构。

```
                     User
                      │
                      ▼
             Workspace（工作台）
                      │
                      ▼
             Workflow Engine
                      │
      ┌───────────────┼───────────────┐
      ▼               ▼               ▼
Research Agent   Script Agent   Publish Agent
      │               │               │
      └───────────────┼───────────────┘
                      ▼
              Plugin System
                      │
      ┌───────────────┼───────────────┐
      ▼               ▼               ▼
    LLM            TTS             Image
      │
      ▼
    Output
```

Atlas Core 不直接调用 AI。

所有能力均通过 Agent 与 Plugin 完成。

---

# 四、系统分层

Atlas 采用六层架构。

```
Presentation Layer

↓

Application Layer

↓

Workflow Layer

↓

Agent Layer

↓

Plugin Layer

↓

Infrastructure Layer
```

---

## 4.1 Presentation Layer（展示层）

负责：

用户交互。

包括：

- Desktop
- Web（未来）
- CLI（未来）

该层不包含业务逻辑。

---

## 4.2 Application Layer（应用层）

负责：

组织用户请求。

例如：

创建任务

开始 Workflow

暂停 Workflow

恢复 Workflow

查看历史任务

该层负责协调系统。

不负责 AI 推理。

---

## 4.3 Workflow Layer（工作流层）

这是 Atlas 的核心。

负责：

定义整个内容生产流程。

例如：

```
Topic

↓

Research

↓

Outline

↓

Script

↓

Narration

↓

Subtitle

↓

Publish
```

Workflow：

只负责：

"下一步应该做什么。"

不负责：

"如何完成。"

---

## 4.4 Agent Layer（Agent 层）

负责：

真正完成任务。

例如：

Research Agent

负责：

资料搜索。

Script Agent

负责：

脚本生成。

Publish Agent

负责：

内容发布。

Agent：

职责单一。

可独立升级。

---

## 4.5 Plugin Layer（插件层）

负责：

提供能力。

例如：

OpenAI

DeepSeek

Ollama

Whisper

FFmpeg

YouTube API

Bilibili API

插件不包含业务逻辑。

只提供能力。

---

## 4.6 Infrastructure Layer（基础设施）

负责：

数据库

缓存

日志

文件

配置

消息队列（未来）

---

# 五、核心运行模型

Atlas 的运行模型如下：

```
用户输入主题

↓

创建 Task

↓

选择 Workflow

↓

Workflow 调度 Agent

↓

Agent 调用 Plugin

↓

生成 Output

↓

Workflow 判断下一步

↓

全部完成

↓

输出 Content Package
```

整个系统始终遵循：

```
Workflow

↓

Agent

↓

Plugin
```

Workflow 不直接调用 Plugin。

Agent 不直接控制 Workflow。

Plugin 不包含业务逻辑。

三者职责清晰。

---

# 六、核心对象

Atlas 运行过程中包含几个核心对象。

## Workspace

用户工作空间。

包含：

项目。

任务。

配置。

资源。

---

## Project

一次内容创作。

例如：

《Toyota 为什么成功》

整个 Workflow 都属于一个 Project。

---

## Task

Workflow 中的一个执行实例。

例如：

Research Task。

Script Task。

Publish Task。

Task 可重试。

---

## Workflow

定义任务执行顺序。

例如：

Research

↓

Script

↓

Narration

↓

Subtitle

Workflow 本身不产生内容。

---

## Agent

完成某项能力。

例如：

Research Agent。

Agent 只做一件事。

---

## Plugin

提供能力。

例如：

DeepSeek。

Whisper。

YouTube。

Plugin 永远不知道：

自己属于哪个 Workflow。

---

## Artifact

Agent 输出物。

例如：

脚本。

字幕。

图片。

视频。

所有内容统一称为：

Artifact。

---

# 七、系统设计原则

整个 Atlas 必须遵守以下原则。

## 单一职责

每个 Agent：

只负责一个能力。

---

## 无状态 Agent

Agent 不保存运行状态。

状态统一由 Workflow 管理。

---

## 插件隔离

Plugin 不允许依赖业务。

---

## Workflow 驱动

所有任务必须由 Workflow 调度。

---

## AI Provider 解耦

Atlas Core 不依赖：

OpenAI。

DeepSeek。

Claude。

任何 Provider。

---

## Human in Loop

任何节点：

都允许：

人工查看。

修改。

重新生成。

---

# 十一、总结

Atlas 的核心思想可以归纳为一句话：

> **Workflow 决定做什么，Agent 决定谁来做，Plugin 决定如何做。**

整个系统围绕这一原则持续演进。

无论未来增加多少模型、Agent、平台和能力，Atlas 的核心架构始终保持稳定。
