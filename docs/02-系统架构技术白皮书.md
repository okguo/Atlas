# 2. Atlas 系统本质（必须重新强调）

Atlas 不是：

- AI工具集合
- Workflow工具
- Agent框架

Atlas 是：

> **以 Project 为核心的数据驱动内容生产系统**

---

# 3. 核心系统运行本质（Runtime Model）

Atlas 的运行不是“调用链”，而是：

> **Project 驱动的状态流转系统**

---

## 3.1 运行核心循环

```
Project Created
    ↓
Workflow Selected
    ↓
Workflow Execution Started
    ↓
Task Generated
    ↓
Agent Executed Task
    ↓
Plugin Called
    ↓
Artifact Produced
    ↓
Artifact Persisted
    ↓
Workflow State Updated
    ↓
Repeat Until Complete
```

---

## 3.2 系统本质公式

> Atlas = Project + Workflow + Task + Agent + Plugin + Artifact

---

# 4. 核心对象模型（强化版）

这是整个系统最重要部分。

---

## 4.1 Object Model（统一数据模型）

```
Workspace
   │
   ▼
Project（核心实体）
   │
   ▼
Workflow（流程定义）
   │
   ▼
Task（执行单元）
   │
   ▼
Agent（执行者）
   │
   ▼
Plugin（能力提供）
   │
   ▼
Artifact（结果产物）
```

---

## 4.2 强约束关系（非常重要）

### Project

- 一切系统的起点
- 所有数据必须归属 Project

---

### Workflow

- 描述“如何生产内容”
- 不保存执行结果

---

### Task

- Workflow 的运行实例
- 可失败 / 可重试 / 可恢复

---

### Agent

- 执行 Task
- 无状态

---

### Plugin

- 提供能力
- 不知道业务存在

---

### Artifact

- 唯一“产物实体”
- 所有输出必须结构化

---

# 6. 核心运行机制（重点）

## 6.1 请求进入流程

```
User Input
   ↓
Project Created / Loaded
   ↓
Workflow Selected
   ↓
Workflow Engine Start
```

---

## 6.2 Workflow执行机制（关键）

Workflow 不执行逻辑，只做：

> DAG 状态机

```
Node1 → Node2 → Node3 → Node4
```

每个 Node = Task

---

## 6.3 Task执行机制

```
Task
  ↓
Resolve Agent
  ↓
Execute Agent
  ↓
Call Plugin
  ↓
Return Artifact
```

---

## 6.4 Agent执行机制（结合03文档）

```
TaskContext
   ↓
Agent
   ↓
Plugin Call
   ↓
Result
   ↓
Artifact
```

---

## 6.5 Plugin执行机制

Plugin 是纯能力层：

```
Agent → Plugin → External API
```

Plugin 不允许：

- 保存状态
- 管理流程
- 参与决策

---

# 7. Project 驱动模型（核心升级）

这是 Atlas 与所有系统最大区别。

---

## 7.1 Project 是第一实体

## 7.2 Project 包含

- Task历史
- Artifact集合
- 版本记录
- 发布记录

---

## 7.3 Project 生命周期

```
Created
  ↓
Active
  ↓
Producing
  ↓
Published
  ↓
Analyzing
  ↓
Iterating
  ↓
Archived
```

---

# 8. 数据流模型（非常关键）

Atlas 的数据流是单向的：

```
Input
 ↓

 ↓
Workflow
 ↓
Task
 ↓
Agent
 ↓

 ↓
```

> 数据只能向下流动，不能反向污染上层

# 10. 技术选型（工程级解释）

## 10.1 后端

Spring Boot

原因：

- 结构清晰
- 易拆模块
- 适合 Workflow Engine

---

## 10.2 数据库

PostgreSQL

原因：

- JSONB支持 Artifact
- 事务支持 Workflow
- 扩展性强

---

## 10.3 缓存

Redis

用途：

- Task状态
- Workflow执行缓存

---

## 10.4 AI层

Provider SPI设计

支持：

- OpenAI
- DeepSeek
- Qwen
- Claude
- Gemini
- Ollama

---

## 10.5 Workflow Engine（关键）

👉 自研

原因：

现有框架无法满足：

- Project绑定
- Artifact追溯
- 人工介入
- 可回放机制

---

## 10.6 Plugin系统

SPI / Adapter模式

---

# 11. 系统关键设计原则（工程约束）

---

## 11.1 Project First

所有数据必须归属 Project

---

## 11.2 Workflow Stateless

只描述流程，不保存状态

---

## 11.3 Agent Stateless

只执行，不记忆

---

## 11.4 Plugin Pure Capability

不包含业务逻辑

---

## 11.5 Artifact Fully Traceable

必须记录：

- 来源Agent
- Prompt
- Model
- 时间
- Workflow节点

---

## 11.6 Everything is Replayable（关键）

任何 Workflow：

可以：

- 重跑
- 回溯
- 从中断恢复

---

# 13. 架构总结（一句话）

> Atlas 的本质是：一个由 Project 驱动，通过 Workflow 编排 Task，由 Agent 执行任务，通过 Plugin 提供能力，最终持续生成 Artifact 的内容生产系统。
